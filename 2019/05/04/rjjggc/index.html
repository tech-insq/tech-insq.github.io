软件架构过程(上)

发表于 2019-05-04 更新于 2021-11-30 分类于 工程 ， 架构 本文字数： 3k 阅读时长 ≈ 3 分钟

支持架构的活动

活动过程

graph LR
   定义最初的范围和情景 --> 引入利益相关者
   引入利益相关者 --> 捕获粗略的关注点
   捕获粗略的关注点 --> 定义架构
   捕获粗略的关注点 -.-> 利益相关者的关注点
   利益相关者的关注点 -.-> 定义架构
   定义架构 -.->|输出| 架构描述
   定义架构 -.->|输出| 指引和约束
   定义最初的范围和情景 -->|输入|范围和情景
   范围和情景 -.-> 定义架构
   定义架构 --> C{决策}
   C{决策} -->|需要骨架|创建骨架系统
   创建骨架系统 -.-> 骨架系统
   C{决策} -->|不需要骨架|终止

活动指导原则

成功的架构定义过程，它必须遵循以下的原则：

  • 利益相关者的关注点驱动。过程必须有效地平衡这些关注点。
  • 它必须促进架构决定、原则和针对这些利益相关者的解决方案本身之间的有效沟通。
  • 架构决定和原则必须贯穿整个生命周期，直到最终部署。
  • 它必须是结构化的，即包括一系列的一个或多个步骤和任务，并且每个步骤都有目的、输入、和输出的清晰定义。
  • 它注重实效
  • 它必须灵活，从而它可以根据实际的情况进行裁剪。
  • 它必须技术无关。
  • 它必须和软件开发过程集成（符合良好的软件工程实践以及质量管理标准）。

过程产物

架构定义的主要目标是创建合理的架构，并对捕获的架构描述的所有元素进行生产和维护方面的管理。

在架构定义的过程，同时也会产生很多架构定义的副产品：

  • 对需求以及过程输入项对澄清。
  • 对利益相关者期望的管理。
  • 对架构可选方案的识别和评估
  • 对架构接受标准的描述
  • 创建一系列设计输入项。

架构定义的细节

架构定义活动

graph LR
  统一输入 --> 确定场景
  确定场景 --> 确定相关的架构样式
  确定相关的架构样式 --> 创建候选架构
  创建候选架构 --> 浏览架构可选项
  浏览架构可选项 --> 和利益相关者评估架构
  和利益相关者评估架构 -->C{评估}
  subgraph 架构评估
  C{评估} -->|可接受|终止
  C{评估} -->|不可接受|重做架构
  重做架构 -.->重新了解需求
  重新了解需求 -.-> 重做架构
  C{评估} -->|不可接受|重新了解需求
  end
  重做架构 -->浏览架构可选项

过程完成标准

原则

一旦完成系统面临的重大风险已经得到缓解，我们就可以认为架构定义已经完成，这可以通过不出现重大意见或者利益相关者对架构评估后的反应来判断。

策略

​ 把你自己包含在架构描述的评审人员名单中，并且直到你对架构满意，认为其中没有重大问题，才能够完成最初的架构描述。

策略

我们的目标是创建足够好、能够满足用户需求的架构描述，而不是非要创建完美的版本，那会花费更多资源才能完成，但不会为系统的利益相关者提供任何真正的好处。

软件开发生命周期中的架构定义

  • 瀑布式方法

  • 迭代方法

    迭代方法背后的动机是通过在早期交付部分功能，从而降低风险。

  • 敏捷方法

    敏捷开发者很少考虑到架构描述，认为它是"大量预先设计"而摒弃。所以好的软件架构师可以做很多事情来确保项目的成功：

    • 以递增的方式交付架构，在早期定义最基本的架构结构。
    • 与团队协同工作以达成一系列清晰的设计原则，并确保大家理解这些原则。
    • 清晰地定义组件，并编写它们的责任和接口文档。
    • 使用简单的方式分享信息，比如wiki。
    • 确保每件交付品都有用户。
    • 把目标设定为创建"足够好"的文档。
    • 如果可以实践的话，和团队一起工作来创建可工作的示例或者模型，验证想法，并指导开发工作中重要或者高风险的地方。
    • 专注于交叉的关注点，并定义合适的系统级策略和解决方案，可以使用视角来辅助。
    • 确保你与每个团队都可以紧密工作，从而促进双向的看法交流。
    • 专注于对架构重要的领域，并将更细节的设计工作留给开发者。

关注点、原则和决定

架构定义对于架构师和利益相关者来说通常都是一个发现的过程。系统的范围和需求是完成架构定义最明显、最需要的内容，但是其他方面的内容对架构定义也一样重要：

  • 业务和IT策略：为组织设定长期的业务和技术优先级和方向，并且包括现在到"目标状态"的路线图。
  • 目标和驱动力：这是促进利益相关者开始项目的最根本问题。
  • 标准和政策： 会制定特定的内容，即组织如何完成业务或者内部运作。
  • 现实中约束： 时间、资金、团队能力、技术的成熟度和缺陷、限制等。

架构关注点：

架构的关注点是利益相关者对架构的需求、目标、约束、意图和期望。

有很多关注点会形成或者影响架构定义，通常它们被归纳为两类：

  • 专注于问题的关注点
  • 专注于解决方案的关注点

专注于问题的关注点

专注于问题的关注点回答的是关于系统"为什么" 和"什么"的问题：为什么需要做某件事或者需要做什么？

它们包括业务策略、业务目标和驱动力、系统范围和需求以及业务标准和政策。

业务策略

业务策略从整体上或者部分定义了业务方向。它会关注以下问题：

  • 提供什么商品和服务，解决了什么问题。
  • 谁是它的客户，客户从中获得了什么好处。
  • 有何新颖，与众不同，如何区别自己和竞争对手。

业务目标和驱动力

业务目标和驱动力为项目设定了业务情景，并且是它存在的根本原因。

定义

​ 业务目标是组织所有的一种特定目的，而业务驱动力是作用于组织的某种力量，需要它以某种特定的方式表现，以保护业务并促进它的增长。

系统范围和需求

系统范围 定义了系统的主要职责，也就是它所需要提供的重要功能。

需求： 功能性需求 和 质量属性

业务标准和政策

业务标准和政策会指定组织如何开展业务或者内部运作的特定内容。它们可能是由规则、已接受的最佳实践或者组织内部的习性和做事方式驱动的。

专注于解决方案的关注点

专注于解决方案的关注点会影响和限制专注于问题的关注点中定义的问题的解决方案。专注于解决方案的关注点会建议或者指定构建系统的特定方式，或者确定和说明如何构建系统，从而对架构产生影响。

技术标准和政策

  • 开放标准，比如ISO、IEEE、W3C。
  • 专属标准，由商业化公司或者其他大型组织创建和管理的。
  • 事实标准，没有得到独立的标准实体的认可，但是得到广泛的遵守。
  • 组织标准，内部组织制定和使用。

其他现实世界中的约束

  • 技术约束
  • 时间
  • 成本
  • 技能
  • 运维约束
  • 物理约束
  • 组织和文化约束

好的关注点

  • 它是定量和可度量，或者满足SMART 原则。
  • 它是可验收，可以通过某种方式测试，从而验证是否达到标准。
  • 它是可跟踪的，可以向后根据策略和目标来验证，也可以向前跟踪到架构或者设计特性中。

架构原则

架构原则

​ 架构原则是对信念、方法或者意图的基本声明，那些内容会指导架构的定义。它可能指的是当前的环境，或者是所需要的将来状态。

好的架构原则

  • 有建设性： 有助于突出问题，驱动架构决定，并确立正确的架构框架。
  • 合理的：由业务驱动力、目标和其他原则所激发的。
  • 表达清晰
  • 可测试
  • 有意义

定义自己的原则

根据组织架构定义实践，定义符合自己、更具体的原则。

架构决定

在确定一种架构决定是否是架构意义，从而需要利益相关者的参与和承认的时候，需要问自己以下问题：

  • 它对系统的功能或者质量属性有重大的影响吗？
  • 它会解决项目面临的重大风险吗？
  • 它对构建系统的时间或者成本有意义吗？
  • 决定或者它的基本原理或者含义是否复杂或者不可预料的吗？
  • 做出这个决定付出了大量的时间和精力吗？
  • 对于决定的重要性是有争议或者涉及政治性问题吗？可能会与企业中喜欢的方法、技术或者厂商有关吗？

使用原则关联关注点和决定

对原则最强大的一种应用就是为你的架构决定提供可跟踪性。可跟踪性来自于使用含义和原理把原则关联在一起，可以通过如下的步骤来实现：

  • 首先确定架构的业务驱动力和目标
  • 使用业务驱动力生成一系列业务原则。
  • 使用业务原则生成一系列技术原则。
  • 使用技术原则生成一系列架构决定。
