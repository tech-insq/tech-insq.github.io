  3. 不需要做全局的并发限制，每台Nginx 实例 做好自己的限流就就可以了（负载均衡、再次Nginx 基于事件驱动）、简单有效。

可测试

方案

  1. 由于Nginx 是基于事件驱动，它将请求转发到后台服务，类似将请求push 到队列，不必堵塞到请求执行完成。

  2. 后台服务处理完之后响应到Nginx，类似ups-thread 执行之后回调。

  3. 线程池采用固定的线程池。

难点

  1. 如何真实Main-thread 如何真实模拟现实的请求情况，一种基于泊松过程的思路（采用随机数）、一种不间断下请求（每达到一定数（200），睡眠5毫秒，这时1秒的理论tps 40000）。
  2. 处理耗时模拟：模拟由于真实的请求多导致变慢，应该是一个基于（concurrency，queueLen）的惩罚函数，最低耗时10毫秒，采用sleep的方式来模拟。

指标解释

  1. RTT = 在队列中的时间 + 处理时间。
  2. doTime = 处理时间。

结论

  1. Vegas 算法相对比较优（可能是测试模型有关），VAMID最差（待改造）。
  2. 队列长度对耗时的影响也很大，处理中的请求数=n * 线程池，那么队列中的剩余请求其耗时要增加(n -1) * doTime，所以并发控制阈值接近真实的并发数最佳（如果不考虑并发多导致变慢），从这个角度来看，测试值70%的利用率说明是有效的。
  3. 除了VAMID，平均RTT相对于理想耗时增加的倍数接～1，说明处理中的请求数～1.5倍的线程池（Fix固定限流，就是采用150的并发限制）。
  4. 测试是否可以慢点，来增加通过率，从目前的测试来看，好像不行（VAMID就是要通过慢点）。

什么是API 网关？

API 网关是一种服务，它是外部使用内部服务的入口。API 网关在整个系统的架构位置如图所示：

API Gateway 是在负载均衡之后，它的主要作用是：

  • 协议转换：外部服务一般是通过基于Https(Http)的RESTful 接口访问内部，而内部服务有可能采用其他协议的微服务框架，比如Dubbo、SOFA、Grpc、RESTful，所以需要协议转换。

  • 请求路由：由于内部服务一般是采用基于服务发现的机制进行互相通信，以及存在网络隔离（基于安全考虑、或者容器化部署）、协议转换，所以需要路由层。

  • 屏蔽内部变化：作为提供给第三方调用的接口，必须保持一定的稳定性，但是内部服务在不停地迭代，所以需要隔离内部变化的反腐层。

  • 组合API：组合API一方面是为了提高用户体验，因为App(Mobile)、第三方、HTML(H5)是属于低性能的互联网环境。而API Gate 跟微服务集群是属于高性能的局域网，所以有必要提供业务组合API来提高性能。组合 API一般基于业务，提供实现某个完整业务活动的API（粗力度），方便外部调用，起到聚合作用。

  • 针对“端”提供专有的API：在API Gateway 消化掉不同端的要求，满足不同端的要求，有可能会减少内部服务设计的复杂性。

  • 实现其他功能：身份验证、访问授权、缓存、服务治理（监控、告警、限流降级、业务可观测）、日志审计。
